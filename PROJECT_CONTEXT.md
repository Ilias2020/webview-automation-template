## Цель проекта

Построить **переиспользуемый template UI-автотестов** для проверки WebView-экранов через браузер Chrome.

Основа идеи:

* тестируется не native-часть приложения
* тестируется web-часть, которая открывается внутри WebView
* запуск идёт через Chrome
* используется mobile layout / mobile viewport

Проект делается как **отдельный framework**, а не как часть mobile Appium framework.

## Что уже решено архитектурно

### Стек

* **Java 17**
* **Gradle (Kotlin DSL)**
* **Selenide**
* **TestNG**
* **Allure**
* **Chrome**

### Почему выбран Selenide

Selenide выбран как основной UI layer, потому что:

* меньше boilerplate-кода, чем в чистом Selenium
* удобные ожидания
* проще Page Object
* быстрее старт framework
* лучше подходит для QA template, чем низкоуровневый Selenium

### Тестовый фреймворк

Используется **TestNG**, не JUnit.

Это важно для:

* аннотаций тестов
* интеграции с Allure
* структуры запуска

## Текущая проблема продукта и временное решение

### Проблема

На dev-окружении логин/UI-авторизация осложняется CAPTCHA.

Из-за этого обычный путь:

* открыть страницу
* выполнить логин внутри автотеста
* перейти в каталог

сейчас неудобен и нестабилен.

### Что уже выяснено

Обычный запуск Selenide/WebDriver:

* создаёт новую WebDriver session
* поднимает отдельный Chrome instance
* не использует вручную открытый браузер пользователя
* не подхватывает автоматически существующую авторизацию

Попытка использовать живой пользовательский Chrome profile напрямую приводила к ошибкам вида:

* `SessionNotCreatedException`
* `Chrome instance exited`

### Временное рабочее решение

Сделан отдельный локальный Chrome user-data-dir для автотестов.

Используется конфигурация:

```properties
chrome.user.data.dir=build/chrome-user-data
chrome.profile.directory=Automation
```

Это позволяет:

* изолировать автотестовый профиль от обычного Chrome
* сохранить локальную авторизованную сессию
* стабильно запускать тесты

### Важное следствие

Нельзя запускать:

```bash
./gradlew clean test
```

потому что `clean` удаляет папку `build`, а вместе с ней и локальный Chrome profile.

### Правильный запуск на текущем этапе

Используется команда:

```bash
./gradlew test --rerun-tasks
```

Причины:

* тест реально выполняется заново
* не удаляется build-профиль Chrome
* сохраняется локальная авторизация

## Текущая конфигурация properties

В `config.properties` используются настройки примерно такого вида:

```properties
browser=chrome
base.url=https://mseller.dev.m-market.kg
browser.size=390x844
headless=false
timeout=10000
page.load.timeout=30000
chrome.user.data.dir=build/chrome-user-data
chrome.profile.directory=Automation
```

### Значение `base.url`

`base.url` хранит **домен**, а не полный URL каталога.

То есть правильно:

```properties
base.url=https://mseller.dev.m-market.kg
```

А конкретные маршруты должны открываться на уровне PageObject, например:

```java
Selenide.open("/catalog?query=&tab=all");
```

## Подход к Page Object

Выбран стиль:

* локаторы хранятся **внутри Page / Component**
* отдельная папка `locators` **не используется**
* используется Selenide-style подход

### Что это значит

Например:

* `SelenideElement` внутри страницы
* `$x(...)` или `$(...)` прямо в PageObject

### Что не используем

* отдельные классы только под локаторы
* fragile-архитектуру в стиле `page → locators`
* избыточный fluent style без необходимости

## Текущая реализованная страница

### `CatalogPage`

Сейчас реализована базовая страница каталога.

На данный момент в ней есть:

* константа маршрута страницы каталога
* открытие каталога
* проверка, что каталог открыт

### Важный архитектурный момент

Метод `open()` приведён к простому action-стилю.

То есть используется:

```java
public void open()
```

а не:

```java
public CatalogPage open()
```

Причина:

* fluent chain пока не нужен
* тесты должны быть простыми и читаемыми
* стиль проекта сейчас: **Page делает действия, Test делает проверки**

## Первый работающий тест

### `CatalogPageTest`

Сейчас реализован первый smoke-тест.

Сценарий:

* открыть страницу каталога
* проверить, что страница каталога открыта

### Важный архитектурный вывод

URL страницы был вынесен из теста в `CatalogPage`.

Это значит:

* тест не знает конкретный маршрут
* маршрут хранится в PageObject
* если URL изменится, правка будет в одном месте

Тест должен оставаться максимально чистым.

Есть:

* открытие отчёта через `allureServe`

* пользовательские `@Step`

### Что уже добавлено в тесты

Используются аннотации:

* `@Description`
* `@Severity`
* `@Story`
* `@Test(description = "...")`

### Что уже добавлено в PageObject

Методы `CatalogPage` помечены через `@Step`, чтобы в отчёте отображались понятные шаги, например:

* «Открыть страницу каталога»
* «Проверить, что страница каталога открыта»

### Почему это важно

Без `@Step` отчёт Allure показывает только сырые действия Selenide.

С `@Step` отчёт становится читаемым для команды.

## Подход к общению и работе в этом проекте

Для этого проекта важны следующие правила:

1. Отвечать кратко, по делу, без воды.
2. Работать строго пошагово.
3. Один шаг → остановка → ожидание подтверждения.
4. Не давать сразу несколько больших шагов.
5. Не писать код без явной необходимости.
6. Сначала логика и архитектура, потом код.
7. Если можно выбрать лучший практический вариант — выбирать его.
8. Не предлагать лишние варианты без необходимости.
9. Существующую рабочую структуру не ломать без причины.
10. Важна стабильность, а не «красота ради красоты».
